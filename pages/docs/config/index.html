--- title: Configuration jade: true toc: true --- h2#introduction Introduction p. Since GLXOSD v3, most of configuration is done through a set of Lua scripts that give almost complete control over how the OSD looks and what information is shown in it. Because you are given the complete Lua language to play with, you may define common constants, dynamically determine values, etc... h2#paths Paths p. By default, GLXOSD looks for configuration scripts in the following directories (in that particular order): ul li $XDG_CONFIG_HOME/glxosd/ (if $XDG_CONFIG_HOME isn't set, the default value of ~/.config/glxosd/ will be used instead) li /etc/glxosd/ li $GLXOSD_SCRIPTS_ROOT/glxosd/conf/ (this is usually /usr/share/glxosd/glxosd/conf/) p. You can modify the path by modifying the content of chainload.lua which can be located in one of the directories mentioned above. h2#common-config-table-types Common configuration table types h3#colors Colors p. Colors are represented by a table with 4 keys: r, g, b and a, which stand for red, green blue and alpha respectively. The values must be numbers between 0 and 1. p. Examples: ul li {r=1, g=0, b=0, a=1} is red. li {r=0, g=1, b=0, a=1} is green. li {r=0, g=1, b=1, a=1} is cyan. li {r=0, g=0, b=0.5, a=0.5} is half-opaque navy. h3#key-combos Key combinations p. Key combinations are represented by a table with 2 keys: ul li main_key – the main key of the key combination. For example, for the combination Shift + F10 the main key is F10. The names for the keys should be as defined in X11/keysymdef.h. li modifiers – a list of modifier keys that have to be pressed when the main key is pressed. Possible elements include: shift, alt, control, caps. p. Examples: ul li p. Shift + F10 pre. { main_key = "F10", modifiers = {"shift"} } li p. Ctrl + Alt + K pre. { main_key = "k", modifiers = {"control", "alt"} } h3#common-markup Common markup options p. The background color of each symbol. tr th.row gamma td. number (positive) td p. Value used for gamma correction. tr th.row underline overline strikethrough td. boolean td p. A table describing the style of an underline/overline/strikethrough. table.table thead.thead-inverse tr th Key th Value type th Value tbody tr th.row enabled td. boolean td p. The color of the underline/overline/strikethrough h2#config-files Config files h3#config-files-glxosd_config_lua glxosd_config.lua p. This is the main configuration file. Once it executes, the following global variables should be set: ul li. GLXOSD_CONFIG – the main configuration table. li. CURRENT_PATH_CONFIG – the table that maps library names to their paths. Please see paths.lua for an example. h3#config-files-paths_lua paths.lua p. The default glxosd_config.lua runs a file called paths.lua first. Its purpose is to help the dynamic linker locate the required shared libraries. p. The default paths.lua contains logic to locate libnvidia-ml on Ubuntu-based systems. h3#config-files-arch_lua arch.lua p. GLXOSD ships with arch.lua, which is a script that determines the architecture of the host system, as opposed to the architecture of the current application. This is useful for determining paths in multiarch systems. h3#config-files-chainload_lua chainload.lua p. chainload.lua is a script that must return the list of configuration paths where to look for further configuration scripts. This essentially allows you to specify custom locations for configuration scripts. h2#globals Configuration globals h3#globals-GLXOSD_CONFIG GLXOSD_CONFIG p. GLXOSD_CONFIG is a global table which contains the following keys: ul li p. plugin – the value should be a list of tables, where each table contains the following keys: ul li path – the path to the plugin's main file. li enabled – set to false to disable the plugin and set to true to enable the plugin. li config – the configuration table for the plugin. h3#globals-CURRENT_PATH_CONFIG CURRENT_PATH_CONFIG p. CURRENT_PATH_CONFIG must contain paths or names to shared libraries to be used by the dynamic linker.
